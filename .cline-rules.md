# 🚀 Cline Full Stack Modern Rules & Best Practices

## 📋 Checklist Pré-Alteração (SEMPRE EXECUTAR ANTES DE EDITAR)

### 1. Análise de Impacto
- [ ] Identificar TODOS os arquivos que serão afetados
- [ ] Verificar dependências do arquivo a ser alterado
- [ ] Confirmar se a alteração não quebrará imports existentes
- [ ] Verificar se há testes relacionados

### 2. Backup de Segurança
- [ ] Fazer commit do estado atual antes de alterações grandes
- [ ] Criar branch para mudanças significativas
- [ ] Documentar mudanças no CHANGELOG.md

### 3. Validação de Tipos
- [ ] Verificar se todos os tipos estão definidos
- [ ] Confirmar compatibilidade de tipos entre frontend/backend
- [ ] Testar build local antes de commit

## 🎯 Regras de Ouro para Full Stack Moderno

### Frontend (React + TypeScript)

#### ✅ SEMPRE:
1. **Usar TypeScript estrito**
   - Sempre definir tipos explícitos para props, state e funções
   - Evitar `any` a todo custo
   - Usar interfaces para objetos complexos

2. **Componentes funcionais com hooks**
   - Usar `useState`, `useEffect`, `useCallback`, `useMemo`
   - Extrair lógica complexa em custom hooks
   - Memoizar componentes pesados com `React.memo`

3. **Tratamento de erros**
   - Usar Error Boundaries para componentes críticos
   - Implementar fallbacks UI para estados de erro
   - Logar erros de forma estruturada

4. **Performance**
   - Lazy loading para rotas e componentes grandes
   - Debounce para inputs de busca
   - Otimizar re-renders com `useCallback` e `useMemo`

5. **Estilização**
   - Usar Tailwind CSS com consistência
   - Criar variantes de componentes com `class-variance-authority`
   - Manter design system coeso

#### ❌ NUNCA:
1. Não mutar estado diretamente
2. Não usar índices como key em listas dinâmicas
3. Não fazer fetch em componentes sem cleanup
4. Não ignorar erros de TypeScript
5. Não criar componentes com mais de 300 linhas

---

### Backend (Node.js + Fastify)

#### ✅ SEMPRE:
1. **Validação rigorosa**
   - Usar Zod para validar todas as entradas
   - Sanitizar dados antes de processar
   - Validar schemas de request/response

2. **Tratamento de erros**
   - Criar classes de erro customizadas
   - Middleware global de tratamento de erros
   - Nunca expor stack traces em produção

3. **Segurança**
   - Usar helmet para headers de segurança
   - Implementar rate limiting
   - Validar JWT em rotas protegidas
   - Hash de senhas com bcrypt

4. **Banco de dados**
   - Usar Prisma para queries type-safe
   - Implementar migrations controladas
   - Usar transações para operações múltiplas

5. **Logging**
   - Logs estruturados com pino
   - Diferentes níveis (info, warn, error)
   - Não logar dados sensíveis

#### ❌ NUNCA:
1. Não concatenar strings em queries SQL
2. Não armazenar senhas em plain text
3. Não expor erros internos do banco
4. Não fazer N+1 queries
5. Não ignorar validação de entrada

## 🚨 Erros Comuns e Como Evitá-los

### 1. Quebra de Imports
**Problema**: Mover/renomear arquivos sem atualizar imports
**Solução**: 
- Usar IDE para refatorar
- Buscar por referências antes de mover
- Testar build após mudanças

### 2. Tipos Incompatíveis
**Problema**: Frontend espera um tipo, backend envia outro
**Solução**:
- Compartilhar tipos entre frontend/backend
- Usar Zod para validar em runtime
- Testar integração

### 3. Memory Leaks
**Problema**: useEffect sem cleanup
**Solução**:
- Sempre retornar cleanup function
- Usar AbortController para fetch
- Cancelar subscriptions

### 4. Race Conditions
**Problema**: Múltiplas atualizações de estado
**Solução**:
- Usar useCallback para funções
- Implementar debounce
- Usar AbortController

### 5. Erros Não Tratados
**Problema**: Promises sem catch
**Solução**:
- Try/catch em async/await
- .catch() em promises
- Error Boundaries
